# Plan: Incremental Fine-Tuning (STT + Translation) with Confidence-Based Correction UI

## Context
The STT application uses Faster-Whisper for inference but has no training capability. The user wants to improve transcription accuracy incrementally by correcting mistakes in the UI and using those corrections as training data. Additionally, Faster-Whisper already supports word-level confidence scores (`word_timestamps`), but they're completely unused ‚Äî we'll leverage these to highlight uncertain words and offer alternative transcriptions, reducing the effort needed to make corrections.

**Key constraint:** Limited GPU (<16GB VRAM), so we must use LoRA/PEFT for memory-efficient fine-tuning.

## Phase 1: Enable Word-Level Confidence Scores & UI Highlighting

### 1.1 Enable `word_timestamps` in Faster-Whisper calls
- **File:** `speech_to_text.py` ~line 1500
- Add `word_timestamps=True` to transcription parameters (both live and file modes)
- Extract per-word data from `seg.words`: each word has `.word`, `.start`, `.end`, `.probability`

### 1.2 Extend database schema
- **File:** `speech_to_text.py` ~line 2133
- Add new table `transcription_words`:
  ```sql
  CREATE TABLE IF NOT EXISTS transcription_words (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      transcription_id INTEGER REFERENCES transcriptions(id),
      word TEXT NOT NULL,
      start_time REAL,
      end_time REAL,
      confidence REAL DEFAULT 1.0,
      corrected_word TEXT DEFAULT NULL,
      FOREIGN KEY (transcription_id) REFERENCES transcriptions(id) ON DELETE CASCADE
  )
  ```
- Add `audio_path TEXT DEFAULT NULL` column to `transcriptions` table (to link audio clips for training)

### 1.3 Store word-level data
- After transcription, save each word with its confidence to `transcription_words`
- Save the audio segment clip (already available from audio_capture.py WAV backup) linked to the transcription

### 1.5 Frontend: Interactive word display
- **File:** `templates/index.html` (and related JS)
- **Every word is clickable** ‚Äî rendered as individual `<span>` elements with click handlers

## Phase 2: Alternative Suggestions for Low-Confidence Words

### 2.1 Generate alternatives via multiple decoding passes
- When a user clicks a word/segment, the backend re-transcribes that audio segment with:
  - Higher `beam_size` (e.g., 10) and `best_of=5`
  - Multiple `temperature` values: [0.0, 0.2, 0.4, 0.6, 0.8]
  - Collect unique alternative transcriptions for that segment
- New API endpoint: `POST /api/get_alternatives` ‚Äî accepts segment ID, returns list of alternative transcriptions ranked by probability

### 2.2 Frontend: Correction popup UI
- Clicking **any word** shows a dropdown/popup with:
  1. **Alternative suggestions** (top 3-5) ‚Äî generated by re-transcribing that segment (fetched on-demand when popup opens)
  2. **Text input field** for manual typing ‚Äî always available as last resort
  3. "Accept" button to save the correction
- Priority order: pick an alternative (one click) > type manually (fallback)
- The same popup works for translation corrections too (shows alternative translations)

## Phase 4: LoRA Fine-Tuning Pipeline

### 4.1 New file: `finetune_whisper.py`
- **Dependencies to add:** `peft>=0.7.0`, `evaluate`, `jiwer` (for WER metric)
- Uses HuggingFace Transformers `WhisperForConditionalGeneration` + PEFT LoRA
- **Important:** Training uses standard PyTorch Whisper (not Faster-Whisper). Faster-Whisper is inference-only (CTranslate2). After training, we convert back to CTranslate2 format for use with Faster-Whisper.
- Key design decisions for 10-12GB VRAM:
  - **Base model: User-selectable** in training settings UI. Options:
    - `openai/whisper-medium` (~6-8GB VRAM for LoRA) ‚Äî safe default
    - `openai/whisper-large-v3` (~10-12GB VRAM with aggressive LoRA r=8, gradient checkpointing, fp16) ‚Äî higher quality, tight fit
    - `openai/whisper-small` (~4GB VRAM) ‚Äî fallback for very limited GPUs
  - **LoRA config:** rank=16 (medium) or rank=8 (large-v3), alpha=32, target modules: `q_proj`, `v_proj` (attention layers)
  - **Training:** fp16 mixed precision, gradient checkpointing, batch_size=2-4 with gradient_accumulation=4-8
  - **Optimizer:** AdamW 8-bit (via `bitsandbytes`) to save memory
  - Load training data from `training_data/manifest.jsonl`
  - Whisper feature extractor + tokenizer for preprocessing
  - Save LoRA adapter weights (small, ~10-50MB) separately from base model

### 4.1b Translation fine-tuning in `finetune_whisper.py`
- Same file handles both model types via a `model_type` parameter
- NLLB LoRA fine-tuning:
  - Base model: `facebook/nllb-200-distilled-600M` (already downloaded in `models/`)
  - ~3-4GB VRAM with LoRA ‚Äî very comfortable on 10-12GB GPU
  - LoRA config: rank=16, alpha=32, target modules: encoder/decoder attention layers
  - Training data: text pairs from `training_data/translation/manifest.jsonl`
  - No CTranslate2 conversion needed ‚Äî NLLB runs in Transformers directly
  - After training, the LoRA adapter is loaded on top of the base NLLB model at inference time

### 4.2 Training configuration in `config.json`
- New section:
  ```json
  "training": {
    "enabled": true,
    "base_model": "openai/whisper-medium",
    "lora_rank": 16,
    "lora_alpha": 32,
    "learning_rate": 1e-4,
    "epochs": 3,
    "batch_size": 4,
    "gradient_accumulation_steps": 4,
    "fp16": true,
    "gradient_checkpointing": true,
    "min_training_samples": 50,
    "confidence_threshold": 0.7,
    "output_dir": "models/whisper-finetuned"
  }
  ```

### 4.3 Separate training server: `training_server.py`
- **Runs as an independent Flask server** on a different port (e.g., 5001) from the main STT server (port 80)
- This keeps training completely isolated ‚Äî no impact on live transcription performance, no GPU contention during training
- **Supports both model types:**
  - **Whisper (STT)** ‚Äî LoRA fine-tuning + CTranslate2 conversion
  - **NLLB (Translation)** ‚Äî LoRA fine-tuning (runs directly in Transformers, no conversion needed). NLLB-200-distilled-600M is only 600M params, very light on VRAM (~3-4GB for LoRA)
- API endpoints on the training server:
  - `POST /api/start_training` ‚Äî starts fine-tuning job. Body: `{"model_type": "stt"|"translation", "base_model": "...", ...}`
  - `GET /api/training_status` ‚Äî returns progress (epoch, loss, ETA, GPU memory usage)
  - `POST /api/stop_training` ‚Äî stops training gracefully
  - `GET /api/training_data/stats?type=stt|translation` ‚Äî returns dataset size, recent corrections
  - `POST /api/convert_model` ‚Äî triggers CTranslate2 conversion (STT only)
  - `GET /api/models?type=stt|translation` ‚Äî lists available fine-tuned models
- WebSocket support for real-time training progress updates
- The main STT server's training UI page proxies/links to the training server
- New page: `templates/training.html` ‚Äî training dashboard with tabs for STT and Translation training
- Can be started/stopped independently: `python training_server.py`

### 4.4 Training schedule & triggers
- **Corrections are saved in real-time** ‚Äî every accepted correction is immediately written to the training dataset (no delay)
- **Training runs after hours** ‚Äî configurable schedule (e.g., nightly at 2am) via cron-style scheduler or manual trigger
- Training server config:
  ```json
  "training_schedule": {
    "enabled": true,
    "cron": "0 2 * * *",
    "min_new_corrections": 10
  }
  ```
- Can also be triggered manually from the training dashboard at any time
- Training server can auto-start/auto-stop: spin up for training, shut down when done (no GPU usage during the day)

### 4.5 Incremental training support
- Save LoRA checkpoints after each training run
- Next training session loads the latest LoRA adapter and continues training with new + old data
- Keep a training history log

## Phase 5: Model Conversion & Deployment

### 5.1 Convert fine-tuned STT model to CTranslate2 format
- After Whisper training completes:
  1. Merge LoRA weights into base Whisper model (`model.merge_and_unload()`)
  2. Convert to CTranslate2 using `ct2-faster-whisper-converter` (ships with faster-whisper)
  3. Save to `models/faster-whisper-finetuned/`
- Add this as an automated post-training step in `finetune_whisper.py`

### 5.1b Deploy fine-tuned translation model
- After NLLB training completes:
  - No conversion needed ‚Äî LoRA adapter is loaded directly on top of the base NLLB model
  - Save adapter to `models/nllb-finetuned-lora/`
  - Modify translation loading code in `speech_to_text.py` to optionally load LoRA adapter via PEFT

### 5.2 Register as custom model on the models page
- Fine-tuned STT models saved to `models/faster-whisper-finetuned-{timestamp}/` appear as selectable custom models on the existing **model manager page** (`templates/model-manager.html`)
- Named with a user-friendly label (e.g., "Custom Fine-tuned (2026-02-15, 150 corrections)")
- Listed alongside the standard Whisper models (tiny, base, small, medium, large-v3, etc.)
- User selects them the same way they'd select any other model ‚Äî no special config needed
- Same for translation: fine-tuned NLLB models appear as selectable translation models
- Multiple fine-tuned versions can coexist (user can switch between them or revert to base)

### 5.3 Model hot-swap
- After training + conversion completes, the new custom model appears on the models page
- User switches to it from the models page like any other model
- Existing model loading code already supports reloading
- Emit WebSocket notification when a new fine-tuned model is ready to use
